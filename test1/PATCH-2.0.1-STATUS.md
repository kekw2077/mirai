### Важная поправка к диагнозу
Прошлая редакция утверждала, что GigaAM, денойз и Piper падали по одной причине —
конфликту `onnxruntime.dll`. Это подтвердилось: после onedir-пересборки все три
поднялись. Но в ходе проверки всплыла ловушка API: у `stt.config` поле
`denoise_dir` — это **корень моделей**, а не папка денойза (`stt_engine._dfn_path()`
сам дописывает `denoise-df/dpdfnet_baseline.onnx`), тогда как `gigaam_dir` и
`voice_dir` — наоборот, конкретные папки. Передашь `denoise_dir` как папку —
получишь «strong denoise model not found» и решишь, что баг не починен.

### Как выпускать следующий патч
1. `pubspec.yaml` — поднять версию; запись в `kChangelog` (main.dart).
2. `flutter build windows --release` (сверить FileVersion exe).
3. `& "C:\Users\ART\AppData\Local\Programs\Inno Setup 6\ISCC.exe" /DAppVersion=X.Y.Z dist\installer.iss`
   → `dist\out\EVS-Setup-X.Y.Z.exe`.
4. `.\dist\sign_update.ps1 .\dist\out\EVS-Setup-X.Y.Z.exe` → length + sha256 +
   dsaSignature → новый `<item>` в начало `dist/appcast.xml`.
5. `gh release create desktop-vX.Y.Z dist\out\EVS-Setup-X.Y.Z.exe --repo kekw2077/mirai ...`
6. Коммит pubspec + main.dart + appcast. **Порядок важен:** сначала релиз, потом
   коммит appcast — иначе клиенты полезут за файлом, которого ещё нет.
   Коммит appcast и есть «включить обновление» для всех.

Сайдкар версионируется независимо от приложения: `build_exe.ps1 -ComponentVersion N`
обновляет `dist/components.json` (sha256+size), затем `gh release upload
desktop-components ... --clobber` и коммит `components.json`. Подпись DSA
компоненту не нужна — он проверяется по sha256.

### §14 — приём голоса по сети (`/command/voice`)
- **Сайдкар (`stt_engine.py`)**: `SttEngine.transcribe_pcm(np, pcm_i16)` — one-shot

### Фикс петли перезапуска после обновления (in-app updater)
**Корень (не WinSparkle — самописный `AppUpdater`):** `applyAndRestart` ставил
инсталлятор без `/DIR`, а `installer.iss` всегда ставит в фиксированный
`{autopf}\EVS`; при этом перезапускался **запущенный** exe
(`resolvedExecutable`). Для портативной копии (напр. `E:\EVS`) новые файлы
уезжали в AppData, запущенная копия не обновлялась, версия не росла → окно
обновления предлагалось на каждом запуске.
- **`applyAndRestart`**: инсталлятору передаётся `/DIR="<папка запущенного exe>"`
  (ставит поверх реально работающей копии) + `/RELAUNCH=1`; убран самоперезапуск
  старого exe — теперь новую версию поднимает сам инсталлятор
  (`[Run] Check: ShouldRelaunch` → `{app}\evs.exe`). Форс-закрытие процессов и
  `/LOG` сохранены.
- **`installer.iss`**: `UsePreviousAppDir=no` — чтобы `/DIR` всегда побеждал
  (иначе Inno берёт папку из реестра по `AppId`, возможно устаревшую AppData).
- **`_checkPreviousUpdateOutcome`**: при успехе (версия совпала с `pending`)
  принудительно показывает окно (`windowManager.show/focus` — overlay иначе его
  прячет) + тост «EVS обновлён до X». При провале запоминает
  `updLastFailedVersion`.
- **Защита от повторной петли**: если фид снова предлагает
  `updLastFailedVersion`, вместо модального «Перезапустить» показывается диалог с
  кнопкой «Скачать» (открывает страницу релиза через `explorer.exe`). Маркер
  снимается при успешной установке.
- **⚠ Миграция:** фикс приходит ВНУТРИ обновления, поэтому текущую «застрявшую»
  копию через сломанный апдейтер не вылечить — нужен **один ручной установ**
  исправленной сборки; дальше обновления работают штатно.
